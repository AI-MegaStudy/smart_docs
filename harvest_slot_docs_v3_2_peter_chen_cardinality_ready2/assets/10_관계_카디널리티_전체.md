# 관계 카디널리티 전체 문서 v3.2

이 문서는 최종 19개 테이블 기준의 전체 관계를 누락 없이 정리한 문서다. 개발자는 이 문서와 `10_어트리뷰트_사전_19테이블.md`만 보고 Miro에서 ERD를 직접 그릴 수 있어야 한다.

## 0. 읽는 기준

```text
From 1개 기준 To 수 = From 엔티티 한 건이 To 엔티티를 몇 건 가질 수 있는가
To 1개 기준 From 수 = To 엔티티 한 건이 From 엔티티 몇 건에 속하는가
FK 컬럼 = 관계선의 자식 쪽 테이블에 존재하는 컬럼
참조 대상 = FK가 참조하는 부모 쪽 PK 또는 UNIQUE 컬럼
```

## 0.1 Peter Chen 카디널리티 표기

이 문서에서는 관계선에 Mermaid/Miro Crow's Foot 기호를 쓰지 않는다.

Miro에서는 관계선 중앙 또는 선 옆에 다음 형식으로 적는다.

```text
관계명(english_relation_name) / Peter Chen 카디널리티
```

| Peter Chen 표기 | 의미 | 이 문서에서의 해석 |
|---|---|---|
| `1:1` | 1대1 관계 | 한쪽 엔티티 1건이 상대 엔티티 최대 1건과 연결된다. 선택성은 `0..1` 또는 `1`로 별도 표기한다. |
| `1:N` | 1대다 관계 | From 엔티티 1건이 To 엔티티 여러 건과 연결될 수 있다. `0..N`이면 선택, `1..N`이면 필수 다건이다. |
| `N:M` | 다대다 업무 관계 | 직접 DB 관계로 두지 않고 연관 엔티티로 해소한다. 이번 ERD의 실제 FK 선은 해소 테이블을 기준으로 `1:N` 또는 `1:1`로 그린다. |

예시:

```text
고객 프로필 확장(extends_customer_profile) / 1:1
예약 요청(requests_reservation) / 1:N
예약 ↔ 수확 슬롯 / N:M -> reservation_items로 해소
```

## R-001 고객 프로필 확장(extends_customer_profile)

- From: `accounts`
- To: `customer_profiles`
- 관계명 라벨: `고객 프로필 확장(extends_customer_profile)`
- Peter Chen 카디널리티 표기: `1:1`
- From 1개 기준 To 수: `0..1`
- To 1개 기준 From 수: `1`
- 분류: 카테고리성 릴레이션십
- FK 컬럼: `customer_profiles.account_id`
- 참조 대상: `accounts.account_id`
- FK 제약: UNIQUE, NOT NULL
- 업무 의미: 계정 하나는 고객 프로필을 없거나 하나만 가질 수 있다. 고객 프로필 하나는 반드시 계정 하나에 연결된다. accounts.role이 CUSTOMER인 경우에만 생성한다.
- Miro 작성법: `accounts` 박스와 `customer_profiles` 박스를 연결하고, 선 라벨을 `고객 프로필 확장(extends_customer_profile) / 1:1` 형식으로 적는다. FK는 `customer_profiles.account_id`에 표시한다.

## N:M 해소 구조

이번 ERD는 직접적인 N:M 관계를 그대로 두지 않고, 다음 연관 엔티티로 해소한다.

| 업무 관계 | 해소 테이블 | 연결 관계 |
|---|---|---|
| 예약 ↔ 수확 슬롯 | `reservation_items` | `reservations -> reservation_items -> harvest_slots` |
| 주문 ↔ 예약 품목 | `order_items` | `orders -> order_items -> reservation_items` |
| 발주 ↔ 주문 품목 | `procurement_items` | `procurements -> procurement_items -> order_items` |

## 선택적 1:1 구현 구조

| 관계 | 구현 방식 |
|---|---|
| `accounts -> customer_profiles` | `customer_profiles.account_id`가 `accounts.account_id`를 참조하며 `UNIQUE, NOT NULL` |
| `accounts -> owner_profiles` | `owner_profiles.account_id`가 `accounts.account_id`를 참조하며 `UNIQUE, NOT NULL` |

| `orders -> procurements` | `procurements.order_id`가 `orders.order_id`를 참조하며 `UNIQUE, NOT NULL` |
| `order_items -> procurement_items` | `procurement_items.order_item_id`가 `order_items.order_item_id`를 참조하며 `UNIQUE, NOT NULL` |

## 배타 카테고리 규칙

`accounts`는 고객과 점주의 공통 상위 엔티티다. DB는 `customer_profiles.account_id`와 `owner_profiles.account_id`를 각각 UNIQUE FK로 관리한다. 같은 계정이 두 프로필을 동시에 갖지 않도록 하는 배타 규칙은 API/서비스 로직에서 강제한다.

```text
accounts.role = CUSTOMER -> customer_profiles 1건 생성 가능, owner_profiles 생성 불가
accounts.role = OWNER    -> owner_profiles 1건 생성 가능, customer_profiles 생성 불가
```
